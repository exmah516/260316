## 我现在理解的目标逻辑（按你最新澄清）

### 1) 轴1蠕动区间端点（不再用“距离值计算”）
- **起始点（StartAbs）**：`StartAbs = rightlimit[轴1] - 56`（这是轴1的绝对位置坐标）。
- **终止点（EndAbs）**：`EndAbs = rightlimit[轴1] - 36`（区间长度 20mm）。
- 触发规则：
  - **手柄回拉**使轴1碰到 **StartAbs** → 轴1 **反向**以最大速度跑到 **EndAbs**。
  - **手柄前推**使轴1碰到 **EndAbs** → 轴1 **反向**以最大速度跑到 **StartAbs**。

### 2) 正常跟随段
- 在区间内，轴1按手柄推拉连续跟随（不提前分段），轴2按手柄旋转跟随。
- 轴3在“正常跟随段”做镜像：轴1怎么动（方向/位移/速度变化趋势），轴3跟同样的相对变化；当轴1进入“快速反向回跳”时，轴3暂停更新，等轴1回跳结束再恢复。

### 3) 手柄开机不再做 Finding Positive/Negative Limit
- 删除 `Finding Positive Limit (Force +1.0) / Finding Negative Limit (Force -1.0)` 这一整段自动推拉标定。
- 保留 b0（0x01）按住回中逻辑，但 **center_pos** 改成“开机取一次当前值/均值”作为中位，不再通过强推强拉寻找。

### 4) b7（0x80）新增：按下脱离控制、松开重同步后恢复
- b7按下：上位机停止写 `G.refer`（以及停止写与运动相关的电缸/fast flag），机器人保持在 PLC 当前接管的参考/状态，不再被手柄影响。
- b7松开：上位机必须先做一次“读回倍福→写回 refer→重置手柄参考点”的**同步**，避免突跳，然后才恢复控制。

### 5) 解决“手柄程序重启后机器人攒动”
- 根因是：上位机启动/恢复控制时，**手柄参考点与 PLC 当前 Act_pos/refer 不一致**，第一帧写 refer 产生阶跃。
- 解决：把“恢复控制”统一走同一个同步流程：
  - 读 `G.Act_pos`（相对）/ `G.init_pos` / `G.rightlimit`
  - 立即把 `pos[0..6] = Act_pos` 写回 `G.refer`（对齐）
  - 再把 `handle_ref/handle_rot_ref/axis*_base_rel` 设为当前值
  - 下一周期开始才允许写入新的跟随目标
